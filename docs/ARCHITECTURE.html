Core Architecture

Breadbrich Engels is a single Node.js process (orchestrator) that:

  1. Polls multiple messaging channels (Telegram via grammy, Slack via Bolt Socket Mode)
  2. Stores all messages in SQLite (store/messages.db)
  3. Dispatches to isolated Docker containers running Claude Code SDK agents
  4. Receives agent output via stdout sentinel markers and file-based IPC
  5. Routes responses back to users and writes KB updates to disk
  6. Exposes a web dashboard at kb.example.com (Express + basic auth, served by separate breadbrich-kb process)

2. Core Orchestrator

Startup Sequence

  1. Ensure Docker daemon running, kill orphaned containers
  2. Initialize SQLite schema (migrations auto-applied)
  3. Load KB permissions (people files → admin/tag cache)
  4. Recover state from DB (cursors, sessions, registered groups)
  5. Start credential proxy on port 3001
  6. Register SIGTERM/SIGINT shutdown handlers
  7. Initialize channels (Telegram, Slack — skip if credentials missing)
  8. Start subsystems: scheduler, IPC watcher, email poller
  9. Recover pending messages from crash
  10. Enter infinite message loop (poll every 2s)

Message Processing Pipeline

  1. Poll: Query DB for new messages since lastTimestamp across all registered groups
  2. Advance cursor: Immediately (crash-safe — prevents reprocessing on restart)
  3. Deduplicate by group: Batch messages per chat JID
  4. Trigger check: Does any message match @Breadbrich Engels pattern AND sender is allowed?
  5. Pipe or enqueue: If container already running for group, pipe via IPC. Otherwise spawn new container.
  6. ACK reaction: Add eyes emoji on receipt, replace with checkmark on completion
  7. Format: Messages formatted as XML with sender, time, reply context
  8. Dispatch: JSON prompt written to container stdin
  9. Stream output: Parse OUTPUT_MARKERS from stdout, send results to user incrementally
  10. Cursor rollback on error: Only if no output was sent to user (prevents reprocessing AND duplicate responses)

Based on node:22-slim with Chromium (browser automation), git, and claude-code SDK installed globally. Agent-runner TypeScript recompiled on every startup (allows per-group customization).

Agent-Runner Inside Container

  1. Read ContainerInput JSON from stdin
  2. Initialize Claude Code SDK with bypassPermissions mode
  3. Start MCP server (IPC tools: send_message, modify_kb_file, schedule_task, etc.)
  4. Enter query loop: send prompt → stream results via OUTPUT_MARKERS → wait for IPC input → repeat
  5. On _close sentinel file, exit gracefully

Output Protocol

---NANOCLAW_OUTPUT_START---
{"status":"success","result":"Agent response text","newSessionId":"abc123"}
---NANOCLAW_OUTPUT_END---

Multiple result chunks emitted per session. Host parses markers from stdout stream.

5. Credential Proxy

HTTP proxy running on the host at 127.0.0.1:3001. Containers route all API calls through it. Real credentials are NEVER passed to containers.

Two Auth Modes

ModeDetectionContainer SendsProxy Injects
API KeyANTHROPIC_API_KEY in .envx-api-key: placeholderReal x-api-key
OAuth (current)No API key, CLAUDE_CODE_OAUTH_TOKEN presentAuthorization: Bearer placeholderReal Bearer token (on exchange request only)

Security Properties

  • Credentials read from .env via readEnvFile() — never loaded into process.env
  • Hop-by-hop headers stripped (connection, keep-alive, transfer-encoding)
  • Upstream URL configurable via ANTHROPIC_BASE_URL (default: api.anthropic.com)
  • Transparent to Claude SDK — same API interface, SDK doesn't know it's proxied

6. IPC System

File-based inter-process communication. Container agents write JSON files to /workspace/ipc/; host orchestrator polls and executes.

Directory Layout

data/ipc/{groupFolder}/
  messages/       # Outbound messages to other channels
  tasks/          # Scheduled task management
  input/          # Host writes here for container to read
    sender_context.json   # KB person context for this request
    _close                # Sentinel: signal container to exit
    {timestamp}.json      # Piped messages from host

IPC Actions

TypeActionAuthorization
messageSend message to any channel/JIDMain: any JID. Others: own group or admin-only for cross-group
emailSend email via SMTPWhitelist:
modify_kb_fileWrite/delete KB files (tasks, people, calendar, etc.)Admin: any. Coordinator: calendar/tasks/artifacts. Others: denied
schedule_taskCreate/modify/delete scheduled tasksNon-main: own group only. Main: any group.
register_groupRegister new chat groupMain group only

Atomic File Writes

IPC files use temp-rename pattern: write to {name}.tmp, then rename() to {name}.json. Prevents host from reading partial files.

8. Identity, RBAC & Permissions

Identity Resolution Pipeline

  1. Get platform ID (Telegram user ID, Slack user ID, CLI username)
  2. Query user_identities table → KB person name

9. Task Scheduler

Schedule Types

TypeExampleBehavior
cron0 9 * * *Standard cron in configured timezone
interval3600000Milliseconds between runs (drift-corrected: anchored to last next_run)
once2026-05-01T09:00:00One-time execution, then completed

Execution Flow

  1. Scheduler polls every 60s for due tasks (status=active AND next_run <= now)
  2. Optional script phase: bash script runs first, output parsed as {wakeAgent: bool, data: any}
  3. If wakeAgent=true (or no script): spawn container with enriched prompt
  4. Result sent to chat_jid and logged in task_run_logs
  5. next_run recomputed; once tasks set to completed

10. Concurrency & Queue Management

GroupQueue State Machine

Each group has a state: {active, idleWaiting, isTaskContainer, pendingMessages, pendingTasks[], retryCount}

  • Concurrency limit: MAX_CONCURRENT_CONTAINERS (default 5)
  • Priority: Tasks > Messages (tasks won't be re-discovered from DB)
  • Retry: Exponential backoff: 5s, 10s, 20s, 40s, 80s (max 5 retries)
  • Idle piping: If container is idle-waiting, new messages piped via IPC file instead of spawning new container
  • Shutdown: Does NOT kill containers — they finish naturally via idle timeout

16. Key Design Decisions

Why cursor-based polling (not webhooks)?

Simple, crash-safe, works across all channels uniformly. Cursor advanced before processing; rolled back only if no output sent (prevents both lost messages and duplicates).

Why file-based IPC (not sockets)?

No socket servers in containers (reduces attack surface). One-directional, atomic (temp-rename), per-group isolated. Host does all authorization checks.

Why credential proxy (not env vars)?

Containers are untrusted. Even if an agent is exploited, it cannot exfiltrate API keys. Proxy holds secrets in host process memory only.

Why per-group container isolation?

Each group is a separate trust domain. Non-main groups can't see other groups' messages, sessions, or IPC. Prevents privilege escalation between Telegram DMs, Slack channels, etc.

Why KB files (not just SQLite)?

Container agents need filesystem access to read/write KB. File-based KB with frontmatter is portable, diffable, and doesn't require DB drivers in containers. Dashboard reads files live from disk.

Why git-pull deploys (not rsync)?

GitHub main is the source of truth. Eliminates "my Mac is the source of truth" fragility. Changes must be committed to survive deploys.

Why named Cloudflare tunnel (not A record)?

No ports exposed to internet. DDoS protection. Permanent URL (kb.example.com) unlike quick tunnels that change on restart.

Naming history

The droplet and system user were renamed from their pre-2026-05-04 names to breadbrich in a single coordinated operation. Transition symlinks remain in place for the legacy /opt/breadbrich, /opt/breadbrich-backups, /opt/breadbrich-git, and /home/breadbrich paths so older scripts still resolve. Upstream package is NanoClaw.
